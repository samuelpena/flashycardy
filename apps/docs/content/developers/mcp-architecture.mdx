# MCP architecture & dependencies

How **`apps/web`** wires Clerk OAuth discovery, the Streamable HTTP MCP endpoint, and tool handlers. For client setup, see [Model Context Protocol (MCP)](/api/mcp/overview).

### Layer summary

| Layer | Path | Role |
|-------|------|------|
| Discovery | `app/.well-known/**` | RFC 9728 / 8414 metadata for MCP OAuth (Clerk handlers). |
| Transport | `app/api/mcp/route.ts` | `mcp-handler` + auth wrapper; exposes GET/POST/DELETE on `/api/mcp`. |
| Auth | `lib/mcp/verify-mcp-token.ts` | Bearer → `AuthInfo` (OAuth first, session JWT fallback). |
| Registry | `lib/mcp/register-tools.ts` | Registers 17 tools; maps `authInfo.extra` → `McpToolContext`. |
| Execution | `lib/mcp/tools/*` | Zod input + `run*` → Drizzle query helpers (`@/db/queries/*`). |

## Third-party connection flow (OAuth + MCP)

How an external MCP host (Cursor, Claude Desktop via `mcp-remote`, custom agent, etc.) discovers Clerk, obtains a token, lists tools, and calls them.

```mermaid
sequenceDiagram
  autonumber
  participant Client as Third-party MCP client
  participant MetaPR as GET /.well-known/oauth-protected-resource/mcp
  participant MetaAS as GET /.well-known/oauth-authorization-server
  participant Clerk as Clerk OAuth / IdP
  participant MCP as POST /api/mcp
  participant Auth as verifyMcpToken
  participant Tools as register-tools + tools/*

  Note over Client,MetaAS: Discovery (RFC 9728 / 8414)
  Client->>MetaPR: Fetch protected resource metadata
  MetaPR-->>Client: resource, authorization_servers, scopes_supported
  Client->>MetaAS: Optional: authorization server metadata
  MetaAS-->>Client: authorize/token endpoints, registration, etc.

  Note over Client,Clerk: OAuth 2.0 (Clerk Dashboard: OAuth app + dynamic client registration if required)
  Client->>Clerk: Register client / authorize user / token exchange
  Clerk-->>Client: OAuth access_token

  Note over Client,MCP: MCP session (Streamable HTTP)
  Client->>MCP: initialize (Authorization: Bearer access_token)
  MCP->>Auth: withMcpAuth → verifyMcpToken(req, token)
  Auth->>Auth: auth(oauth_token) + verifyClerkToken
  alt OAuth valid
    Auth-->>MCP: AuthInfo { extra: userId, hasUnlimitedDecks }
  else Session JWT fallback
    Auth->>Auth: authenticateRequest(session_token)
    Auth-->>MCP: AuthInfo { extra: userId, hasUnlimitedDecks }
  end
  MCP-->>Client: initialize result (server capabilities)

  Client->>MCP: tools/list
  MCP-->>Client: list_decks, create_deck, … (17 tools)

  Client->>MCP: tools/call (e.g. list_decks)
  MCP->>Tools: requireToolContext → runListDecks(ctx, args)
  Tools->>Tools: @/db/queries/decks (scoped by ctx.userId)
  Tools-->>MCP: JSON text { data, meta }
  MCP-->>Client: tool result
```

### MCP methods the client typically uses

| Phase | MCP / HTTP | What happens |
|-------|------------|----------------|
| Discovery | `GET` well-known URLs | Client learns Clerk issuer, scopes (`profile`, `email`), and resource identifier for `/api/mcp`. |
| OAuth | Clerk-hosted | User consents; client receives **OAuth access token** (recommended). |
| Session | `POST /api/mcp` | `initialize` — `withMcpAuth` runs **`verifyMcpToken`** before handler body runs. |
| Tools | `tools/list` | **`registerFlashycardyMcpTools`** exposes 17 tool names + Zod-shaped inputs. |
| Invoke | `tools/call` | Handler reads **`parseMcpAuthExtra`**, calls **`run*`** with **`McpToolContext`**, returns REST-shaped JSON in text content. |

### Auth decision inside `verifyMcpToken`

```mermaid
flowchart TD
  A[Bearer token present?] -->|no| U[undefined → 401 via withMcpAuth]
  A -->|yes| B[auth acceptsToken oauth_token]
  B --> C[verifyClerkToken]
  C -->|userId in extra| D[AuthInfo + hasUnlimitedDecks from oauthAuth.has]
  C -->|no| E[authenticateRequest session_token]
  E -->|signed-in| F[AuthInfo + hasUnlimitedDecks from session has]
  E -->|not signed-in| U
```
